<task type="auto">
  <name>Task 1: Pin report period boundaries</name>
  <files>tests/test_executive_report.py, app.py</files>
  <action>
    Extend `tests/test_executive_report.py` with focused tests for `report_period_dates()` and, where useful, `report_period_label()`.

    Cover these cases:
    - `period="day"` returns the fixed current day as both start and end.
    - default/unknown week behavior returns the last seven days through the fixed current day, preserving the current inclusive behavior rather than converting it to calendar weeks.
    - a past month such as `2026-01` returns the full month, including the real last day.
    - the current month returns the first day through the fixed current day, not the future end of the month.
    - a past year such as `2025` returns January 1 through December 31.
    - the current year returns January 1 through the fixed current day.

    Control "today" without adding dependencies. Prefer monkeypatching `app.date` with a small `date` subclass whose `today()` returns a fixed `date(2026, 5, 14)`, and keep assertions independent of the machine clock.

    If a test exposes an actual period-boundary bug, make the narrowest correction in `app.py` around `report_period_dates()`, `normalize_report_month()`, or `normalize_report_year()`. Do not change the week business rule unless the test proves current code contradicts existing documented behavior.
  </action>
  <verify>python -m pytest -q tests/test_executive_report.py</verify>
  <done>Report period tests pass deterministically with a fixed current date and document the current day/week/month/year boundaries.</done>
</task>
